# 正则表达式引擎（NFA/DFA 双模式）系统设计文档

## 一、项目概述

### 1.1 项目背景
正则表达式是文本检索、日志过滤、词法解析场景的核心工具，底层依托有限自动机实现字符串匹配。主流正则引擎分为 **NFA（非确定有限自动机）与DFA（确定有限自动机）** 两种实现方案：NFA 易于实现、原生支持捕获分组，但存在回溯带来的灾难性回溯性能缺陷；DFA 无回溯、匹配效率稳定，需要通过 NFA 子集构造 + 状态最小化得到最简自动机。

本项目从零实现一套兼容 POSIX 接口的正则引擎，完整实现正则表达式→抽象语法树 AST→ε-NFA→DFA→最小化 DFA全链路编译流程，支持常用正则语法、捕获分组、NFA/DFA 双模式匹配，并与 Linux 系统POSIX regex.h库做功能兼容性、性能对比测试，深入理解正则引擎底层原理与自动机优化技术。

### 1.2 功能需求
（1）正则语法支持
- 元字符：`.`任意字符、`^`行首锚、`$`行尾锚；
- 量词：`*`、`+`、`?`、`{m,n}`区间重复；
- 分支：`|`多选分支；
- 字符类：`[]`正向字符集、`[^]`反向字符集、预定义类`\d`、`\w`、`\s`；
- 语法：`()`括号分组 + 捕获组，记录每组匹配起止下标。

（2）核心算法实现
- 正则词法语法解析，构建抽象语法树 AST；
- Thompson 构造算法：AST 转换为带 ε 空转移的 NFA，支持状态转移表打印、DOT 可视化导出；
- 子集构造算法：NFA 转换为 DFA；
- Hopcroft 算法实现 DFA 状态最小化（本项目预留实现接口，可扩展完成）；

（3）上层 POSIX 兼容 API
- `regex_compile`：正则编译，支持 NFA/DFA 两种编译模式；
- `regex_match`：全文精准匹配；
- `regex_search`：子串检索，从文本任意位置匹配；
- `regex_findall`：全局非重叠匹配，返回所有匹配结果及捕获组信息；

（4）测试与性能要求
- 功能：通过 Grep 开源正则测试集 90% 以上用例；
- 性能：DFA 模式下匹配效率不低于 POSIX 标准正则库的 80%；
- 交付物：NFA/DFA 状态转移表、自动机可视化 DOT 示例图、多场景性能对比数据表。

### 1.3 开发环境
- 操作系统：Ubuntu Linux
- 开发语言：标准 C 语言（C99）
- 编译工具：GCC
- 可视化工具：Graphviz（解析 DOT 文件绘制 NFA/DFA 状态图）

## 二、总体架构设计

### 2.1 分层架构
本系统采用四层分层架构，自上而下依次为：API 调用层→AST 语法层→NFA 自动机层→DFA 自动机层，层间低耦合、依赖单向调用。
┌─────────────────────────────────────┐
│ 上层API调用层（regex.c） │
│ regex_compile / regex_match / search │
│ findall 、结果封装、匹配逻辑调度 │
└───────────────────┬─────────────────┘
↓
┌─────────────────────────────────────┐
│ AST语法解析层（parser+ast） │
│ 正则字符串词法扫描 → 语法分析 → 生成AST │
└───────────────────┬─────────────────┘
↓
┌─────────────────────────────────────┐
│ NFA自动机层（nfa.c） │
│ Thompson构造、ε闭包计算、状态转移管理 │
│ NFA状态表打印、DOT图导出、捕获组记录 │
└───────────────────┬─────────────────┘
↓
┌─────────────────────────────────────┐
│ DFA自动机层（dfa.c） │
│ 子集构造、DFA状态管理、Hopcroft最小化 │
│ DFA转移表打印、可视化DOT导出 │
└─────────────────────────────────────┘

## 三、详细模块设计

### 3.1 AST 抽象语法树模块

#### 3.1.1 设计目的
消除正则表达式语法歧义，统一结构化存储正则语义，为 Thompson 构造提供标准化输入，区分普通字符、量词、分支、字符类、锚点、分组等语法单元。

#### 3.1.2 核心数据结构
- **AstType**：枚举所有 AST 节点类型，包含字符、任意字符、首尾锚、预定义字符类、拼接、分支、量词重复、捕获组、自定义字符类；
- **CharRange**：存储字符类的连续区间（如`a-z`）；
- **CharClass**：存储自定义字符类，包含是否取反、离散字符数组、连续区间数组；
- **AstNode**：AST 通用节点，使用联合体适配不同类型节点的数据存储。

#### 3.1.3 对外接口
| 接口函数 | 功能说明 |
| :--- | :--- |
| `ast_new_char` | 普通字符节点创建 |
| `ast_new_any_char` | `.`任意字符节点 |
| `ast_new_digit/word/whitespace` | `\d`/\w`/\s`预定义类节点 |
| `ast_new_start/end` | `^`/`$`锚点节点 |
| `ast_new_concat` | 表达式拼接节点 |
| `ast_new_alternation` | `\|` 分支选择节点 |
| `ast_new_repeat` | 量词重复节点（min 最小次数、max 最大次数，SIZE_MAX代表无限） |
| `ast_new_group` | `()`捕获分组节点 |
| `ast_new_char_class` | `[]`字符类节点 |
| `ast_free` | 递归释放整棵 AST 内存，防止内存泄漏 |

### 3.2 正则语法解析模块（Parser）

#### 3.2.1 设计目的
对正则字符串做词法扫描、递归下降语法分析，按照优先级：括号 > 量词 > 拼接 > 分支构建合法 AST。

#### 3.2.2 语法优先级规则
1. 原子项（括号、字符、`.`、转义字符、字符类）优先级最高；
2. 量词`*` `+` `?` `{m,n}`绑定最近的原子项；
3. 表达式拼接（相邻表达式隐式连接）优先级次之；
4. `|`或分支优先级最低。

#### 3.2.3 核心解析流程
- `parse_atom`：解析最小原子单元（括号、字符、转义、字符类、锚点）；
- `parse_factor`：原子项后绑定量词；
- `parse_term`：连续多个因子做拼接运算；
- `parse_expr`：多个项做或分支运算，生成顶层 AST；

### 3.3 NFA 自动机模块

#### 3.3.1 设计目的
基于 Thompson 构造算法，将 AST 转化为带 ε 空转移的非确定有限自动机；支持 ε 闭包求解、状态表可视化、捕获分组状态记录，作为 DFA 转换的输入源。

#### 3.3.2 核心数据结构
- **NFAState**：NFA 状态，包含状态 ID、是否为接收态；
- **TransitionType**：转移类型：ε 空转移、普通字符、任意字符、`\d`/\w`/\s`、自定义字符类；
- **NFATransition**：转移存储联合体，兼容各类字符匹配规则；
- **NFAEdge**：状态有向边：起始状态、转移规则、目标状态；
- **NFA**：NFA 总结构体：动态状态数组、边数组、起始态、接收态、捕获分组状态数组。

#### 3.3.3 核心算法与接口
- `nfa_from_ast`：Thompson 构造入口，递归遍历 AST 生成 NFA 片段，通过串联、并联、闭包规则拼接自动机；
    - **拼接**：前一个表达式终止态 ε 转移到后一个表达式起始态；
    - **分支**：新建总起始态分别 ε 跳转到各分支起始，各分支终止态 ε 汇总到总终止态；
    - **闭包***：支持 0 次直接跳转、多次循环跳转；`+`至少 1 次；`?`0 或 1 次；
- `nfa_epsilon_closure`：DFS 求解 ε 闭包，是 NFA 转 DFA 的核心基础函数；
- `nfa_print_transition_table`：格式化打印所有状态转移关系；
- `nfa_to_dot`：生成 Graphviz DOT 描述字符串，用于绘制 NFA 状态流程图；

#### 3.3.4 捕获组设计
遇到`()`分组时，记录分组内部表达式的起始 NFA 状态、终止 NFA 状态，在文本匹配过程中，通过状态路径记录每组匹配的文本偏移位置。

### 3.4 DFA 自动机模块

#### 3.4.1 设计目的
通过子集构造算法消除 NFA 的不确定性，得到 DFA 确定有限自动机，同一输入字符仅存在唯一后继状态，消除回溯；支持状态最小化压缩冗余状态，提升匹配效率。

#### 3.4.2 核心数据结构
- **DFAState**：DFA 状态，状态 ID、是否接收态；
- **DFA**：动态存储 DFA 所有状态、转移边、起始态、接收态集合。

#### 3.4.3 核心接口
- `dfa_from_nfa`：子集构造算法，以 NFA 的 ε 闭包作为 DFA 初始状态，BFS 遍历所有 NFA 状态集合生成 DFA；
- `dfa_minimize`：Hopcroft 状态最小化算法（预留扩展接口），通过分区精化合并等价状态，压缩状态数量；
- `dfa_print_transition_table`：DFA 状态转移表格式化输出；
- `dfa_to_dot`：DFA 可视化 DOT 文件生成；

#### 3.4.4 关键优化点
原生 NFA 仅能处理枚举字符，本设计扩展支持任意 ASCII 字符集，兼容`.`、`\d`、`[]`等字符类正则的 DFA 转移生成，解决字符类正则 DFA 模式匹配失效问题。

### 3.5 上层引擎 API 模块

#### 3.5.1 设计目的
封装底层 NFA/DFA 细节，对外提供简洁、兼容 POSIX 规范的调用接口，支持两种匹配模式，统一封装匹配结果与捕获组数据。

#### 3.5.2 核心结构体
- **RegexMode**：模式枚举：NFA 回溯模式、DFA 确定模式；
- **RegexMatch**：单次匹配结果：匹配起止偏移、捕获分组数量、每组起止位置数组；
- **RegexMatches**：全局批量匹配结果集合；
- **Regex**：正则引擎上下文，存储原始正则串、编译后的 NFA、DFA 实例、编译标记。

#### 3.5.3 核心匹配逻辑
- **NFA 匹配**：维护当前可达状态集合，逐个读取文本字符求解后继状态 +ε 闭包，贪心匹配最长子串；
- **全文匹配`regex_match`**：从文本 0 偏移开始，必须消耗全部文本到达接收态；
- **检索匹配`regex_search`**：遍历文本所有起始偏移，只要存在任意位置匹配成功即返回；
- **全局匹配`regex_findall`**：循环检索非重叠匹配，收集所有结果；

## 四、核心算法设计

### 4.1 Thompson 构造算法
- **基础规则**：每个原子表达式构建仅包含一条转移的二元状态片段；
- **拼接规则**：表达式 A 结尾 ε 指向表达式 B 开头；
- **选择规则**：总起始态分别 ε 指向 A、B 开头，A、B 结尾分别 ε 指向总结束态；
- **克林闭包`A*`**：总起始可直接 ε 到总结束（0 次），也可进入 A 循环多次匹配；
- **`A+`**：至少匹配一次，无直接 0 次跳转；
- **`A?`**：0 次或 1 次可选匹配。

### 4.2 ε-闭包算法
采用深度优先遍历：
1. 初始化将初始状态集合入栈、标记已访问；
2. 遍历当前状态所有 ε 类型边，未访问的后继状态加入集合与栈；
3. 循环直到栈为空，最终集合即为该组状态的 ε 闭包。

### 4.3 子集构造算法（NFA→DFA）
1. 计算 NFA 起始状态的 ε 闭包作为 DFA 初始状态；
2. 采用 BFS 遍历每个 DFA 状态（对应一组 NFA 状态集合）；
3. 对每个输入字符，收集所有 NFA 可达后继状态，再求 ε 闭包得到新的 NFA 集合；
4. 若该 NFA 集合未生成 DFA 状态，则新建 DFA 状态并入队，否则复用已有 DFA 状态；
5. 记录 DFA 状态转移关系，标记包含 NFA 接收态的 DFA 状态为接收态。

### 4.4 Hopcroft DFA 最小化算法（扩展设计）
1. 初始分区：划分为接收状态集、非接收状态集；
2. 不断对每个分区按照输入字符的后继状态所属分区做细化拆分；
3. 直到所有分区无法再拆分，每个分区内的状态互为等价状态；
4. 每个分区保留一个状态，重构最简 DFA 转移表。
